# Solar Panel Verification Workflow ‚Äì Overpass¬†API Approach

We‚Äôll shift from the unreachable PDOK WFS to a low‚Äëlatency, open source that reliably covers the Netherlands: **OpenStreetMap via Overpass¬†API**. OSM already contains many rooftop solar installations crowdsourced by mappers; extraction is a single HTTP query‚Äîno DNS issues, no zip downloads.

---

## Why Overpass¬†API?

- ‚úÖ¬†**Global Coverage**¬†‚Äì includes North¬†Holland without regional throttling.
    
- ‚úÖ¬†**Simple HTTP GET** ‚Äì works behind most firewalls.
    
- ‚úÖ¬†**Direct GeoJSON** ‚Äì no need for complex parsing.
    
- ‚ùï¬†Completeness varies by area, but North¬†Holland (Amsterdam region) is well mapped.

## 2025‚Äë05‚Äë11¬†Simplified City‚Äëby‚ÄëCity Workflow (using **bag‚Äënh.gpkg**) üü¢

### Performance note ‚Äì JSON vs¬†CSV for final step

- **GeoJSON (or OSM‚ÄëJSON)** is only needed while we carry geometry for the spatial intersect. Once the `has_panels` flag is derived, we no longer need geometries.
    
- **CSV** writes fastest, uses the least RAM, and opens instantly in Excel ‚Äë‚Äë ideal for the **final deliverable**.
    
- Therefore: **keep solar geometries in GeoJSON/GPKG only until the intersect is done**, then export the merged table to CSV.
    
- If you ever need to reload into QGIS, you can still drag‚Äëdrop the CSV (it autogeocodes from the Longitude / Latitude columns).
